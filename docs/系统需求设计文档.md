## 一、项目概述

### 1.1 项目背景

WorkMatch 是一个**蓝领 RPO 招聘管理系统**（内部工具），用于管理蓝领招聘全流程。系统支持招聘交付闭环，并扩展到零工平台生态。

**核心业务特点**：
- 一个候选人可以参与多个项目（跨项目流转）
- 支持人力池复用（失败后可再分配到其他项目）
- 每个项目有独立的状态流转和跟进记录
- 支持特殊项目环节（京东购包/培训、酒店保洁计件等）
- 需要统计招聘漏斗转化率（到面率、到训率、入职率等）

### 1.2 项目定位

| 定位 | 说明 |
|------|------|
| 系统类型 | 内部工具 |
| 用户规模 | 10-50 人团队 |
| 数据量级 | 10万+ 候选人记录 |
| 部署方式 | 单租户，私有部署 |

### 1.3 目标用户

| 角色 | 描述 | 典型操作 |
|------|------|----------|
| 招聘专员 | 一线招聘人员 | 录入候选人、跟进、推进状态 |
| 招聘主管 | 团队管理者 | 查看团队数据、分配任务 |
| 系统管理员 | 系统维护者 | 账号管理、系统配置 |

## 二、业务规则定义（关键）

> ⚠️ 这是蓝领 RPO 业务的核心规则，必须严格遵守

### 2.1 候选人与项目的关系

**规则**：一个候选人可以同时或先后参与多个项目，支持人力池复用。

```
候选人张三:
├── 京东项目: 已报名 → 已邀约 → 待面试 → 已面试 → 试工中 → 失败(未购包)
│
├── 盒马项目: 已报名 → 已邀约 → 待面试 → 已面试 → 试工中
│
└── 酒店保洁项目: 已报名 → 等待邀约（24元/间）
```

**每个项目独立记录**：
- 状态流转历史
- 跟进记录
- 状态变更时间
- 酒店保洁专用：已完成间数

### 2.2 状态流转规则（8 个核心状态）

**完整状态流**：

```
已报名 ──→ 已邀约 ──→ 待面试 ──→ 已面试 ──→ 试工中 ──→ 试工通过
   │          │          │          │          │
   │          │          │          │          └──→ 失败（标签: 未购包/未到训/放弃/不符合要求/其他）
   │          │          │          │
   │          │          └──────────┼──→ 失败（标签: 爽约）
   │          │                     │
   │          └─────────────────────┼──→ 失败（标签: 拒绝）
   │                                │
   └────────────────────────────────┴──→ 失败（标签: 联系不上/其他）
```

**状态说明**：

| 状态 | 代码 | 说明 | 触发条件 |
|------|------|------|----------|
| 已报名 | `signed_up` | 初始状态 | 录入候选人时 |
| 已邀约 | `invited` | 已发起邀约 | 手动推进 |
| 待面试 | `interview_pending` | 约定面试时间 | 手动推进 |
| 已面试 | `interviewed` | 完成面试 | 手动推进 |
| 试工中 | `in_trial` | 正在试工 | 手动推进 |
| 试工通过 | `trial_passed` | 试工合格，已入职 | 手动推进 |
| 失败 | `failed` | 通用失败状态 | 手动推进 + 选择原因标签 |
| 联系不上 | `unreachable` | 无法联系 | 手动推进 |

### 2.3 失败原因标签

| 标签 | 代码 | 适用项目 | 说明 |
|------|------|----------|------|
| 爽约 | `no_show` | 通用 | 约面未到 |
| 拒绝 | `rejected` | 通用 | 候选人拒绝 |
| 未购包 | `no_package` | 京东 | 未购买配送装备 |
| 未到训 | `no_training` | 通用 | 未参加培训 |
| 改约 | `rescheduled` | 通用 | 更改约定时间 |
| 放弃 | `abandoned` | 通用 | 个人放弃 |
| 不符合要求 | `not_qualified` | 通用 | 未通过考核 |
| 其他 | `other` | 通用 | 其他原因（需备注） |

### 2.4 人力池复用规则

**规则**：进入"失败"或"联系不上"状态的候选人，可重新分配到其他项目。

```
京东项目失败(未购包) ──→ 人力池 ──→ 分配到盒马项目（状态重置为"已报名"）
```

**实现方式**：
1. 人员表增加 `reusable` 字段（是否可复用，默认 true）
2. 提供再分配 API：`POST /persons/{id}/reassign`
3. 再分配时创建新的 `person_project` 记录，状态初始化为"已报名"

### 2.5 转化率计算公式

**招聘漏斗指标**：

```
邀约率 = 已邀约人数 / 已报名人数 × 100%
到面率 = 已面试人数 / 待面试人数 × 100%
爽约率 = 爽约人数 / 待面试人数 × 100%
试工率 = (试工中 + 试工通过 + 失败)人数 / 已面试人数 × 100%
入职率 = 试工通过人数 / 已报名人数 × 100%
失败率 = 失败人数 / 已报名人数 × 100%（按标签细分）
```

**特殊项目指标（京东）**：

```
到训率 = 到训人数 / 已面试人数 × 100%（通过扩展字段统计）
购包率 = 已购包人数 / 到训人数 × 100%
```

**酒店保洁专用指标**：

```
打扫率 = 已完成间数 / 分配间数 × 100%
```

### 2.6 时间统计要求

每个状态变更必须记录：
- 变更时间（`changed_at`）
- 变更人（`changed_by`）
- 前一状态（`from_status`）
- 后一状态（`to_status`）

用于统计：
- 平均邀约时长（报名→邀约）
- 平均到面时长（邀约→面试）
- 平均入职周期（报名→入职）

### 2.7 灵工发放流程（预留接口，MVP 后实现）

```
1. 收集四要素（姓名、身份证、手机、银行卡）
2. 导入任务结算单
3. 通知个人完成小程序认证 + 签约
4. 系统发放
```

**预留接口**：
```python
def process_payment(person_id: int):
    info = get_person_four_elements(person_id)
    sync_to_finance(info)
    import_task_settlement()
    notify_person_certify()
    execute_payment()
```

### 5.5 数据统计模块

#### 5.5.1 统计维度

| 维度 | 说明 |
|------|------|
| 按项目 | 各项目的转化率对比 |
| 按时间 | 日/周/月趋势 |
| 按负责人 | 各负责人的业绩对比 |
| 按来源 | 各渠道的效果对比 |
| 按失败标签 | 失败原因细分 |

#### 5.5.2 统计指标

| 指标 | 计算公式 | 说明 |
|------|----------|------|
| 报名数 | COUNT(已报名) | 总报名人数 |
| 邀约数 | COUNT(已邀约及以上) | 已发起邀约的人数 |
| 邀约率 | 邀约数 / 报名数 | 邀约效率 |
| 待面试数 | COUNT(待面试) | 等待面试人数 |
| 已面试数 | COUNT(已面试及以上) | 实际面试人数 |
| 到面率 | 已面试数 / 待面试数 | 邀约质量 |
| 爽约率 | 爽约人数 / 待面试数 | 爽约比例 |
| 入职数 | COUNT(试工通过) | 成功入职人数 |
| 入职率 | 入职数 / 报名数 | 整体转化率 |

#### 5.5.3 功能点

| 功能 | 描述 | 优先级 |
|------|------|:------:|
| 项目统计 | 单项目的漏斗数据 | P1 |

```json

  "message": "success",
```

```json
    "items": [...],
    "total": 100,

    "page_size": 20
  }
}
```
